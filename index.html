Why the Proxy? (vs. Direct Cline + OpenRouter)

Hard spending capWhatever OpenRouter chargesPrepaid credits — when they run out, everything stops

Local modelsNot possible with OpenRouter setupOne-line .env change — zero cost, absolute privacy

The direct setup is perfectly fine. The proxy is an upgrade — easy model switching via .env, cost protection, better privacy, and the flexibility to use local models when you want complete air-gapped coding. It takes 10 minutes to set up and you never have to think about it again.

Step 1: Set Up Your OpenRouter Account (The Safe Way)

Platform-independent — works the same on all OSes via your browser.

  1. Sign up at openrouter.ai and enable 2FA.
  2. Create an API key:
    • Give it a label like ultra-lite-proxy (optional)
    • Credit Limit: $20 per month
    • Reset Limit: Monthly
    • Expiration: Never
  3. Copy the key (sk-or-v1-...).
  4. Go to Billing → ensure Auto Top-Up is disabled.
  5. Purchase $20 in credits (one-time, no recurring).
  6. Guardrails: In Settings → Privacy → Guardrails, create a budget with a $20 monthly cap. Enable Prompt Injection Detection set to "Flag", and Sensitive Info Detection: only "Credit Card" and "SSN" checked (set to "Redact"). Add a custom pattern for your API key (sk-or-v1-[a-zA-Z0-9]{32,}) and set it to Block.
  7. Under Settings → Privacy, ensure "Zero Data Retention" or all training/logging toggles are off.

Now you have a single API key, a hard spending cap, and full privacy controls.

Step 2: Project Directory & Environment

We'll store the proxy — and your API key — in a single global location. No per‑project .env files. No risk of accidentally committing secrets. The repo includes an .env.example — copy it to .env, add your key, and you're done. The only file your projects ever need is a .clinerules.

Why this is safer than per-project .env files: Your API key lives in a single global location (~/.config/ultra-lite-proxy/.env), not scattered across every project folder. That means:

✅ You never accidentally commit it to a project repo
✅ You never need to add .env to every project's .gitignore
✅ One chmod 600 / icacls protects it everywhere
✅ You update your key in one place, and every project picks it up

The proxy is the only thing that reads this file. Cline connects to the proxy. Your projects don't need to know anything about API keys — they just work.

Step 3: The Unified Proxy

The proxy is the heart of the setup — a single Python script that routes your requests to OpenRouter with mode prefix support. All model IDs can be overridden in .env — if a model changes on OpenRouter, just update the corresponding line and restart the proxy. No Python editing needed.

Download the complete proxy script:

View proxy.py on GitHub — Download raw file

Save it inside your ultra-lite-proxy folder. The script includes all fixes from real-world testing:

  • Provider filtering — ignores flaky providers, retries on 502 errors
  • Reasoning disabled on DeepSeek — prevents empty tool-call responses

Using Your Own Models (Ollama, vLLM, LM Studio, etc.)

The LLM Gateway pattern is provider-agnostic. You're not locked into OpenRouter. If you have your own models — whether that's Ollama on a Mac, vLLM on a server rack, LM Studio on a desktop, or any other tool that exposes an OpenAI-compatible API — your proxy can route to them the same way.

What changes in .env

Just point the proxy at your local endpoint and switch the backend:

# .env — using your own models instead of OpenRouter
LLM_BACKEND=local                                    # Switch from "openrouter" to "local"
LOCAL_API_URL=http://localhost:11434/v1              # Ollama default. Change to your endpoint.
# OPENROUTER_API_KEY=sk-...                          # Not needed for local models

# --- Local Models (Ollama, vLLM, LM Studio, etc.) ---
DEFAULT_MODEL=qwen2.5-coder:7b
PLANNING_MODEL=qwen2.5-coder:14b
BULK_MODEL=qwen2.5-coder:7b
RING_MODEL=deepseek-r1:7b
🔒 Privacy by design: The proxy script itself defaults to local-only mode — if LLM_BACKEND isn't set, no data leaves your machine. The guide sets it to openrouter because that's what we're here to set up, but you can switch to local at any time. One word in .env, and your data never leaves your network.

Direct Provider Access

The proxy also supports routing directly to a single provider (like OpenRouter, OpenAI, Anthropic) without model switching — just a transparent passthrough with no .env-based model switching or spending caps. This is useful for testing or when you need to bypass the proxy's routing logic entirely.

# .env — direct provider access
        LLM_BACKEND=direct                                   # Passthrough mode — no routing, no spending cap
        DIRECT_API_URL=https://api.anthropic.com/v1          # Any OpenAI-compatible endpoint
        DIRECT_API_KEY=sk-ant-...                            # Your provider API key
        DIRECT_API_PROVIDER=anthropic                        # "anthropic", "openai", or "groq"
⚠️ Warning: LLM_BACKEND=direct bypasses all proxy safeguards — no .env-based model switching, no provider filtering, and no spending cap. You are billed directly by the provider you point DIRECT_API_URL at. This is not the same as OpenRouter's prepaid system. Use with caution and monitor your usage.
🔒 Privacy advantage: When using local models, your code never leaves your network. No API keys, no external providers, no data retention concerns. Combined with the proxy binding to 127.0.0.1, this gives you a completely air-gapped AI coding assistant if needed. For companies working on sensitive data, this means AI-assisted coding without anything ever leaving the building. The same rules, the same workflow — just running on your own hardware.

Globally install the MCP tools. Open a fresh terminal (not in venv):

If the command isn't available, manually create/edit the file at the path for your OS:

Paste this configuration (adjust the filesystem path to your projects folder):

{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/your/projects"]
    },
    "memory": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-memory"]
    },
    "sequential-thinking": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-sequential-thinking"],
      "disabled": true
    },
    "docs-mcp-server": {
      "type": "sse",
      "url": "http://localhost:6280/sse"
    }
  }
}

Important: Replace /path/to/your/projects with your actual projects folder. Examples:

  • Windows: D:\\Code\\Source
  • macOS: /Users/yourname/Projects
  • Linux: /home/yourname/Projects
⚠️ A word of caution on MCP servers and token costs: Every active MCP server injects its tool definitions into Cline's system prompt, which adds tokens to every single request — even ones that don't use the tool. When a tool is used, its output (often verbose or containing image data) gets added to the conversation history, further bloating the context window.

I originally had Puppeteer (for screenshots) and Gitingest (for repo analysis) in this setup. Puppeteer caused model failures on complex tasks and each screenshot added image data to the context. Gitingest pulled entire repositories into the prompt — a single analysis cost me nearly $6, almost half of what I spent building an entire production app. I removed both.

The four servers above are what I kept: filesystem and memory are essential, docs-mcp-server fetches docs on demand without bloat, and sequential-thinking is disabled by default because it's only useful for complex debugging sessions. Be selective with MCP servers. Each one you add should justify its token cost. When in doubt, start minimal and add only what you genuinely need.

🧠 Context Management — How Cline Tasks Actually Work

Cline tasks persist between sessions. Close Cursor at night, reopen in the morning, and your conversation is still there — no command needed. This is the key to keeping costs low.

🔄 How Tasks Persist

What you doWhat happens
Close Cursor at nightYour task is saved automatically. No action needed.

Return after a week offSame — reopen and continue. No command needed.
Return after a few monthsTry reopening. If it feels sluggish or confused, run /newtask once. It'll create a handover summary and start fresh.
Start a completely different projectRun /newtask. The old task is archived with its handover. The new task is clean.

💰 Why This Matters for Cost

Every time you start a brand new task (not via /newtask), Cline sends its full system prompt, your .clinerules, and MCP tool definitions — about 20–30K tokens — at full price. When you stay in one task, prompt caching kicks in and you pay only ~10% of that for every subsequent request.

StrategySystem Overhead (10 prompts)Total Cost
New task for every prompt10 × 25K = 250K tokensHigh
One task, stay in it1 × 25K + 9 × 2.5K = 47.5K tokens~80% less
One task, /smol every 10–15 messagesSame as above + less conversation bloatLowest

🧠 The golden rule: One project = one task. Stay in it. /smol every 10–15 messages. /newtask only for genuine context switches. Close Cursor when you're done — your task will be waiting for you tomorrow.
