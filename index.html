Step 1: Set Up Your OpenRouter Account (The Safe Way)

Platform-independent — works the same on all OSes via your browser.

  1. Sign up at openrouter.ai and enable 2FA.
  2. Create an API key:
    • Give it a label like ultra-lite-proxy (optional)
    • Credit Limit: $20 per month
    • Reset Limit: Monthly
    • Expiration: Never
  3. Copy the key (sk-or-v1-...).
  4. Go to Billing → ensure Auto Top-Up is disabled.
  5. Purchase $20 in credits (one-time, no recurring).
  6. Guardrails: In Settings → Privacy → Guardrails, create a budget with a $20 monthly cap. Enable Prompt Injection Detection set to "Flag", and Sensitive Info Detection: only "Credit Card" and "SSN" checked (set to "Redact"). Add a custom pattern for your API key (sk-or-v1-[a-zA-Z0-9]{32,}) and set it to Block.
  7. Under Settings → Privacy, ensure "Zero Data Retention" or all training/logging toggles are off.

Now you have a single API key, a hard spending cap, and full privacy controls.

Step 2: Project Directory & Environment

We'll store the proxy — and your API key — in a single global location. No per‑project .env files. No risk of accidentally committing secrets. The only file your projects ever need is a .clinerules.

Why this is safer than per-project .env files: Your API key lives in a single global location (~/.config/ultra-lite-proxy/.env), not scattered across every project folder. That means:

✅ You never accidentally commit it to a project repo
✅ You never need to add .env to every project's .gitignore
✅ One chmod 600 / icacls protects it everywhere
✅ You update your key in one place, and every project picks it up

The proxy is the only thing that reads this file. Cline connects to the proxy. Your projects don't need to know anything about API keys — they just work.

Step 3: The Unified Proxy

The proxy is the heart of the setup — a single Python script that routes your requests to the right AI model based on hashtags and prefixes. All model IDs can be overridden in .env — if a model changes on OpenRouter, just update the corresponding line and restart the proxy. No Python editing needed.

Download the complete proxy script:

View proxy.py on GitHub — Download raw file

Save it inside your ultra-lite-proxy folder. The script includes all fixes from real-world testing:

  • Safe message handling — extracts text without mutating Cline's message history
  • Expanded connection pool — handles Cline's concurrent requests without socket drops
  • Provider filtering — ignores flaky providers, retries on 502 errors
  • Reasoning disabled on DeepSeek — prevents empty tool-call responses
  • Full model list exposed — Cline sees all available models
  • Prompt caching enabled — massive cost savings on repeated context

Platform-independent — Cline config works the same on all OSes.

Install the Cline extension (saoudrizwan.claude-dev) in Cursor/VS Code.

Cline settings (gear icon):

  • API Provider: OpenAI Compatible
  • Base URL: http://127.0.0.1:8000/v1
  • API Key: not-needed (the proxy ignores it; must be non-empty)
  • Model ID: leave blank (proxy overrides)

Model Configuration:

  • Context Window: 1048576
  • Supports Images: Yes
  • Browser: Yes
  • Prompt Caching: No (handled via proxy)
  • Input/Output Price: leave as 0

Feature Settings:

  • Enable: Subagents, Native Tool Call, Parallel Tool Calling, Strict Plan Mode, Auto Compact, Focus Chain, Checkpoints, Cline Web Tools
  • Disable: Yolo Mode, Hooks (unless needed)
  • MCP Display Mode: Plain Text

Terminal Settings: Default profile — pick your OS default (PowerShell on Windows, zsh or bash on macOS/Linux), Terminal Execution Mode VS Code Terminal, Aggressive terminal reuse off.

Browser Settings: Keep browser tool enabled. Uncheck "Use remote browser connection" (this would expose your real Chrome profile). After saving, in the "Available Tools" panel, set the browser tool to "Ask Every Time".

# Project Guidelines for AI Assistants

## General Conduct
- Always ask for clarification if a request is ambiguous. Never assume.
- Prefer simple, readable solutions over clever, complex ones.
- Never add a new dependency without asking first.

## Coding Standards
- Write unit tests for all new public functions.
- Prefer functional, declarative patterns over imperative ones.
- Use meaningful variable and function names.

## Workflow
- For complex, multi-file changes, first use plan mode to outline the architecture.
- After any file change, ensure the project still compiles and all existing tests pass.
- After any database schema change, run `npx payload generate:types` and `npx tsc --noEmit`.
- After any collection or global config change, update `scripts/seed.ts` to match.

# Next.js & Clean Code Architecture Rules

## Core Principles
- **Empathy over Cleverness:** Write code that is easy to read and maintain for humans, not just machines.
- **Keep it Simple:** Avoid over-engineering. Do not add abstractions before they are truly needed.

## 1. Zero Hardcoded or Magic Values
- **Constants:** Move all arbitrary strings, IDs, limits, configurations, and magic numbers into a central file.
- **Enums & Objects:** Group related configuration values into `as const` objects or standard TypeScript enums.
- **Env Variables:** Access all API routes, base domains, and public keys exclusively via `process.env`.
- **No Inline Magic Numbers:** Never use raw numbers in logic calculations or conditionals.
- **No Arbitrary Styling:** Avoid hardcoded Tailwind arbitrary values. Use global design tokens or theme scales.

## 2. Directory Structure & Code Sharing
- **Path Aliases:** Always use configured path aliases (`@/components/`, `@/utils/`) instead of relative paths.
- `src/app/`: File-system routing, layouts, and page entry-points only.
- `src/components/`: Reusable, isolated presentation components.
- `src/helpers/` or `src/utils/`: Pure functions with single, isolated responsibilities.
- `src/services/`: Centralised data-fetching layer, API callers, and third-party wrappers.
- `src/hooks/`: Reusable custom React hooks.

## 3. SOLID & Clean Code Rules
- **Single Responsibility (SRP):** One file, one job.
- **Open/Closed (OCP):** Build components to handle variations using React composition (`children` prop).
- **Dependency Inversion (DIP):** Pass callback behaviors as props instead of embedding logic.
- **Function Length:** Keep methods under 25 lines.

## 4. Next.js & TypeScript Standards
- **Server-First:** Default to React Server Components. `'use client'` only at the lowest node possible.
- **Strict Typing:** Never use `any`. Define clear, strict `interface` properties.
- **Data Fetching:** Isolate error bounds with clean `try/catch` and Next.js `error.tsx` boundaries.
